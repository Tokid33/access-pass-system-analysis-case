# Data Model

## 1. Назначение модели

Модель данных предназначена для хранения заявок на пропуск сотрудников и посетителей, их текущих статусов, истории изменений и информации о пользователях системы.

---

## 2. Подход к моделированию

В модели выделены:
- основная сущность заявки;
- справочные сущности;
- сущность истории изменений;
- сущности пользователей и ролей.

Текущее состояние заявки хранится в сущности `Request`, а история изменений — в сущности `RequestHistory`.

---

## 3. Сущности

## 3.1. Role
Сущность хранит роли пользователей системы.

**Поля:**
- `role_id` — уникальный идентификатор роли
- `role_name` — наименование роли

**Примеры значений:**
- Initiator
- Approver
- Security
- Administrator

---

## 3.2. User
Сущность хранит пользователей системы.

**Поля:**
- `user_id` — уникальный идентификатор пользователя
- `full_name` — ФИО пользователя
- `email` — email пользователя
- `role_id` — ссылка на роль пользователя
- `is_active` — признак активности пользователя

---

## 3.3. PassType
Сущность хранит типы пропусков.

**Поля:**
- `pass_type_id` — уникальный идентификатор типа пропуска
- `pass_type_name` — наименование типа пропуска

**Примеры значений:**
- Employee
- Visitor
- Temporary

---

## 3.4. Request
Сущность хранит основную информацию о заявке на пропуск.

**Поля:**
- `request_id` — уникальный идентификатор заявки
- `request_number` — бизнес-номер заявки
- `pass_type_id` — ссылка на тип пропуска
- `initiator_id` — ссылка на пользователя-инициатора
- `visitor_full_name` — ФИО посетителя / сотрудника, на которого оформляется пропуск
- `visitor_document` — документ, удостоверяющий личность
- `visit_date` — дата визита
- `comment` — комментарий к заявке
- `current_status` — текущий статус заявки
- `rejection_reason` — причина отклонения
- `created_at` — дата и время создания
- `updated_at` — дата и время последнего изменения

---

## 3.5. RequestHistory
Сущность хранит историю изменений заявки.

**Поля:**
- `history_id` — уникальный идентификатор записи истории
- `request_id` — ссылка на заявку
- `changed_by_user_id` — ссылка на пользователя, внёсшего изменение
- `field_name` — изменённое поле
- `old_value` — старое значение
- `new_value` — новое значение
- `changed_at` — дата и время изменения

---

## 4. Ключи

### Primary Key (PK)
- `Role.role_id`
- `User.user_id`
- `PassType.pass_type_id`
- `Request.request_id`
- `RequestHistory.history_id`

### Foreign Key (FK)
- `User.role_id` → `Role.role_id`
- `Request.pass_type_id` → `PassType.pass_type_id`
- `Request.initiator_id` → `User.user_id`
- `RequestHistory.request_id` → `Request.request_id`
- `RequestHistory.changed_by_user_id` → `User.user_id`

---

## 5. Связи между сущностями

- Одна роль может быть назначена многим пользователям (`Role 1:N User`)
- Один пользователь может создать много заявок (`User 1:N Request`)
- Один тип пропуска может использоваться во многих заявках (`PassType 1:N Request`)
- Одна заявка может иметь много записей истории (`Request 1:N RequestHistory`)
- Один пользователь может внести много изменений в историю (`User 1:N RequestHistory`)

---

## 6. Бизнес-логика, отражённая в модели

### 6.1. Текущий статус
Текущий статус заявки хранится в поле `Request.current_status`.

### 6.2. История изменений
Каждое изменение ключевых полей заявки фиксируется в `RequestHistory`.

### 6.3. Причина отклонения
Если заявка переведена в статус `Rejected`, поле `rejection_reason` должно быть заполнено.

### 6.4. Инициатор заявки
Каждая заявка должна иметь одного инициатора, связанного с сущностью `User`.

---

## 7. Допущения модели

- В рамках MVP данные посетителя хранятся внутри сущности `Request`, без выделения отдельной сущности `Visitor`.
- Ролевая модель пользователей поддерживается через сущности `User` и `Role`.
- Текущий статус хранится в заявке как отдельное поле для быстрого доступа, а история статусов — в `RequestHistory`.

---

## 8. Возможные расширения модели

На следующих этапах модель может быть расширена:
- отдельной сущностью `Visitor`;
- отдельным справочником `Status`;
- сущностью `Attachment` для хранения файлов;
- сущностью `Notification` для хранения уведомлений;
- интеграцией с внешними пропускными системами.
