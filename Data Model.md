# Data Model — Access Pass System

## 1. Назначение документа

Этот документ описывает логическую модель данных для системы обработки заявок на пропуск.

Цель модели данных:
- зафиксировать основные сущности предметной области;
- показать связи между ними;
- определить состав ключевых атрибутов;
- объяснить, как аналитические требования переводятся в структуру данных.

---

## 2. Примечание по реализации

Текущая физическая реализация схемы описана в файле `sql/schema.sql`.

В реализованной версии:
- текущий статус заявки хранится в поле `current_status_id`;
- статусы вынесены в справочник `status`;
- история изменения статусов хранится в таблице `request_status_history`;
- история реализована как переходы между статусами (`from_status_id` -> `to_status_id`), а не как универсальный журнал изменения любых полей;
- при переводе заявки в статус `REJECTED` причина отклонения обязательна.

Важно: в рамках текущего MVP в базе данных реализована именно история смены статусов. Полный универсальный аудит изменения любых полей может быть следующим этапом развития модели.

---

## 3. Бизнес-контекст

Система предназначена для централизованного управления заявками на пропуск.

Она должна поддерживать:
- создание заявки;
- хранение информации о посетителе;
- назначение типа пропуска;
- отслеживание текущего статуса;
- фиксацию истории обработки;
- привязку действий к пользователям и ролям;
- возможность дальнейшего анализа данных через SQL.

---

## 4. Основные сущности

### 4.1. UserRole

**Назначение:**  
Справочник ролей пользователей системы.

**Примеры ролей:**
- Initiator
- Approver
- Security
- Administrator

**Атрибуты:**
- `role_id` — идентификатор роли
- `role_name` — наименование роли

---

### 4.2. AppUser

**Назначение:**  
Пользователь системы, который может создавать заявки, согласовывать их или менять статусы.

**Атрибуты:**
- `user_id` — идентификатор пользователя
- `full_name` — ФИО пользователя
- `email` — email пользователя
- `role_id` — ссылка на роль пользователя
- `is_active` — признак активности пользователя
- `created_at` — дата и время создания записи

**Связи:**
- каждый пользователь принадлежит одной роли;
- один пользователь может создать много заявок;
- один пользователь может выполнить много изменений статуса.

---

### 4.3. PassType

**Назначение:**  
Справочник типов пропусков.

**Примеры значений:**
- Employee
- Visitor
- Temporary

**Атрибуты:**
- `pass_type_id` — идентификатор типа пропуска
- `pass_type_name` — наименование типа пропуска

---

### 4.4. Status

**Назначение:**  
Справочник статусов жизненного цикла заявки.

**Примеры значений:**
- CREATED
- IN_REVIEW
- APPROVED
- REJECTED
- COMPLETED
- CANCELLED

**Атрибуты:**
- `status_id` — идентификатор статуса
- `status_code` — системный код статуса
- `status_name` — отображаемое наименование статуса

**Комментарий:**  
Статус используется в двух местах:
- как текущий статус заявки;
- как состояние в истории переходов.

---

### 4.5. PassRequest

**Назначение:**  
Основная бизнес-сущность системы — заявка на пропуск.

**Атрибуты:**
- `request_id` — идентификатор заявки
- `request_number` — уникальный номер заявки
- `pass_type_id` — тип пропуска
- `initiator_id` — пользователь, создавший заявку
- `current_status_id` — текущий статус заявки
- `visitor_full_name` — ФИО посетителя
- `visitor_document` — документ посетителя
- `visit_date` — дата визита
- `comment` — комментарий по заявке
- `rejection_reason` — причина отклонения
- `created_at` — дата и время создания заявки
- `updated_at` — дата и время последнего изменения

**Бизнес-смысл:**
- одна запись = одна заявка на пропуск;
- заявка всегда имеет один актуальный текущий статус;
- при отклонении должна быть указана причина;
- заявка связана с инициатором, типом пропуска и историей статусов.

---

### 4.6. RequestStatusHistory

**Назначение:**  
История переходов заявки между статусами.

**Атрибуты:**
- `history_id` — идентификатор записи истории
- `request_id` — ссылка на заявку
- `from_status_id` — предыдущий статус
- `to_status_id` — новый статус
- `changed_by_user_id` — пользователь, выполнивший изменение
- `comment` — комментарий к изменению
- `changed_at` — дата и время изменения

**Бизнес-смысл:**
- каждая запись фиксирует один переход заявки между статусами;
- история нужна для прозрачности, аудита и аналитики;
- по истории можно восстановить путь обработки заявки.

---

## 5. Связи между сущностями

### UserRole → AppUser
- одна роль может быть назначена многим пользователям;
- каждый пользователь имеет одну роль.

**Тип связи:** 1:N

---

### AppUser → PassRequest
- один пользователь может создать много заявок;
- каждая заявка имеет одного инициатора.

**Тип связи:** 1:N

---

### PassType → PassRequest
- один тип пропуска может использоваться во многих заявках;
- каждая заявка имеет один тип пропуска.

**Тип связи:** 1:N

---

### Status → PassRequest
- один статус может быть текущим статусом для многих заявок;
- каждая заявка имеет один текущий статус.

**Тип связи:** 1:N

---

### PassRequest → RequestStatusHistory
- одна заявка может иметь много записей истории;
- каждая запись истории относится к одной заявке.

**Тип связи:** 1:N

---

### Status → RequestStatusHistory
- один статус может встречаться как исходный или целевой во многих переходах;
- каждая запись истории хранит переход между статусами.

**Тип связи:** 1:N по `from_status_id` и 1:N по `to_status_id`

---

### AppUser → RequestStatusHistory
- один пользователь может выполнить много изменений статуса;
- каждая запись истории фиксирует одного исполнителя изменения.

**Тип связи:** 1:N

---

## 6. Жизненный цикл заявки

Базовый жизненный цикл заявки в кейсе:

- `CREATED`
- `IN_REVIEW`
- `APPROVED`
- `REJECTED`
- `COMPLETED`
- `CANCELLED`

### Общая логика
- после создания заявка получает начальный статус `CREATED`;
- далее заявка может перейти в статус согласования;
- по результату рассмотрения заявка либо одобряется, либо отклоняется;
- после исполнения переводится в `COMPLETED`;
- в отдельных сценариях может быть отменена.

### Важное правило
При переводе заявки в статус `REJECTED` поле `rejection_reason` обязательно.

---

## 7. Ключевые бизнес-правила модели

### BR-1. Один актуальный статус
В каждый момент времени заявка должна иметь только один текущий статус.

**Как отражено в модели:**  
Поле `current_status_id` в `pass_request`.

---

### BR-2. История переходов должна сохраняться
Каждое изменение статуса должно оставлять след в истории.

**Как отражено в модели:**  
Таблица `request_status_history`.

---

### BR-3. Причина отклонения обязательна
Заявка не может быть отклонена без указания причины.

**Как отражено в модели:**  
Поле `rejection_reason` + триггер в `schema.sql`.

---

### BR-4. Роли влияют на действия
Права на изменение статусов зависят от роли пользователя.

**Как отражено в модели:**  
Справочники `user_role`, `app_user`.

**Комментарий:**  
Ограничение ролей на уровне действий зафиксировано как бизнес-правило и требование. В текущей физической схеме оно не реализовано отдельными DB-ограничениями.

---

### BR-5. Заявка должна быть анализируема
Модель должна позволять строить выборки по статусам, датам визита, инициаторам и истории обработки.

**Как отражено в модели:**  
Структура `pass_request`, `request_status_history`, справочники и индексы.

---

## 8. Логика хранения истории

В текущей реализации история организована как **история статусов**, а не как общий аудит всех полей.

Это означает:

**Что фиксируется:**
- по какой заявке произошло изменение;
- из какого статуса в какой;
- кто изменил;
- когда изменил;
- какой комментарий оставил.

**Что пока не фиксируется как отдельная универсальная история:**
- изменение любого произвольного поля заявки;
- старое и новое значение каждого поля;
- тип операции в формате generic audit log.

Для портфельного кейса и MVP-реализации такой уровень детализации достаточен, потому что он покрывает основной бизнес-сценарий — управление жизненным циклом заявки.

---

## 9. Индексы и производительность

В физической схеме предусмотрены индексы по основным аналитическим и операционным сценариям:

- `initiator_id`
- `current_status_id`
- `visit_date`
- `created_at`
- `request_id` в истории статусов
- `changed_at` в истории статусов

Это позволяет ускорить:
- фильтрацию списка заявок;
- поиск по текущему статусу;
- выборки по дате визита;
- анализ истории изменений;
- построение отчётности.

---

## 10. Примеры вопросов, которые покрывает модель

Модель данных позволяет ответить на вопросы:

- сколько заявок сейчас находится в каждом статусе;
- какие заявки были отклонены и по какой причине;
- кто чаще всего создаёт заявки;
- кто и когда менял статус заявки;
- сколько времени заявка находилась в процессе;
- какие заявки назначены на определённую дату визита;
- согласован ли текущий статус заявки с её историей.

---

## 11. Ограничения текущей модели

Текущая модель не покрывает:

- универсальный аудит изменения всех полей;
- отдельную сущность уведомлений;
- отдельную матрицу допустимых переходов статусов;
- хранение вложений/документов;
- интеграцию с внешней СКУД;
- техническую модель API и событий.

Это осознанное ограничение MVP-кейса.

---

## 12. Возможные расширения

Модель можно развить в следующих направлениях:

### 12.1. AllowedStatusTransition
Отдельная таблица допустимых переходов статусов:
- `from_status_id`
- `to_status_id`
- `allowed_role_id`

Это позволит хранить правила переходов в данных.

---

### 12.2. Notification
Отдельная сущность уведомлений:
- тип уведомления;
- получатель;
- канал;
- статус отправки;
- дата отправки.

---

### 12.3. Generic Audit Log
Универсальная история изменений:
- имя поля;
- старое значение;
- новое значение;
- пользователь;
- дата изменения.

---

### 12.4. Attachment / Document
Хранение связанных документов:
- файл;
- тип документа;
- ссылка на заявку;
- дата загрузки;
- автор загрузки.

---

## 13. Итог

Текущая модель данных покрывает ключевой сценарий системы обработки заявок на пропуск:

- есть основная сущность заявки;
- есть ролевая модель пользователей;
- есть справочники типов пропуска и статусов;
- есть текущий статус;
- есть история переходов;
- есть основа для аналитики и SQL-проверок.

Для портфельного кейса это достаточная и целостная модель, которая показывает переход от бизнес-проблемы к структуре данных и исполняемой SQL-реализации.
