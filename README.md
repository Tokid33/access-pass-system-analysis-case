# Access Pass System Analysis Case

## Обзор

Этот репозиторий содержит компактный кейс по системному анализу процесса оформления пропусков.

Кейс показывает, как бизнес-проблема переводится в:
- описание процесса,
- требования,
- модель данных,
- SQL-артефакты,
- воспроизводимую локальную среду PostgreSQL.

Центральный сценарий кейса — жизненный цикл заявки на пропуск: создание, согласование, отклонение, выпуск пропуска и хранение истории изменения статусов.

---

## Проблема

Заявки на пропуск подаются через разные каналы: мессенджеры, почту, устные договорённости.

Это приводит к следующим проблемам:
- заявки теряются;
- нет единого статуса и прозрачности процесса;
- возникают ошибки в данных;
- администратор тратит много времени на ручную координацию;
- участники процесса не понимают, на каком этапе находится заявка.

---

## Цель

Создать централизованную систему управления заявками на пропуск, которая:

- сокращает время обработки заявки;
- делает процесс прозрачным для всех участников;
- снижает количество ошибок;
- фиксирует историю изменения статусов;
- даёт основу для операционной и аналитической отчётности.

---

## Краткий обзор решения

Система включает следующие логические блоки:

- форма подачи заявки;
- backend-логика обработки;
- модуль согласования;
- управление статусами;
- уведомления участников процесса;
- хранение истории изменений.

Основной поток процесса:

**Employee → Manager → Pass Administrator → Issued**

---

## Роли

- **Employee** — создаёт заявку на пропуск;
- **Manager** — согласует или отклоняет заявку;
- **Pass Administrator** — выпускает пропуск и завершает обработку.

---

## Scope

В рамках кейса рассматриваются:

- создание заявки на пропуск;
- согласование заявки;
- отклонение заявки;
- изменение текущего статуса;
- хранение истории смены статусов;
- базовые уведомления;
- базовая модель данных;
- SQL-запросы для проверки и анализа процесса.

---

## Out of Scope

В кейс не входят:

- полноценный UI/UX интерфейс;
- production-ready backend;
- авторизация и разграничение доступа на уровне приложения;
- интеграции с реальными системами безопасности;
- реальная отправка email / SMS / push-уведомлений;
- нефункциональная проработка enterprise-уровня.

## Модель данных

Текущая физическая реализация схемы описана в файле:

- [sql/schema.sql](./sql/schema.sql)

Основные сущности:

- `pass_request` — заявка на пропуск;
- `request_status_history` — история изменения статусов заявки;
- `status` — справочник статусов;
- `pass_type` — справочник типов пропусков;
- `app_user` — пользователи системы;
- `user_role` — роли пользователей.

Модель поддерживает:

- хранение текущего статуса заявки;
- фиксацию переходов между статусами;
- хранение причин отклонения;
- привязку заявки к инициатору;
- базовую аналитическую обработку через SQL.

---

## Что реализовано в SQL-части

В репозитории реализованы:

- схема PostgreSQL;
- тестовые данные;
- набор SQL-запросов для операционных и аналитических сценариев.

Примеры задач, которые можно проверить через SQL:
- список заявок с текущими статусами;
- отклонённые заявки и причины отклонения;
- история изменения статусов;
- количество заявок по статусам;
- анализ времени обработки заявки.
