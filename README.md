# Access Pass System Analysis Case

## Обзор

Этот репозиторий содержит компактный кейс по системному анализу процесса оформления пропусков.

Кейс показывает, как бизнес-проблема переводится в:
- описание процесса;
- требования;
- модель данных;
- SQL-артефакты;
- воспроизводимое локальное окружение PostgreSQL.

Центральный сценарий кейса — жизненный цикл заявки на пропуск: создание, согласование, отклонение, выпуск пропуска и хранение истории изменения статусов.

---

## Проблема

Заявки на пропуск подаются через разные каналы: мессенджеры, почту, устные договорённости или таблицы.

Это приводит к следующим проблемам:
- заявки теряются;
- нет единого статуса и прозрачности процесса;
- возникают ошибки в данных;
- администратор тратит много времени на ручную координацию;
- участники процесса не понимают, на каком этапе находится заявка;
- сложно формировать отчётность и контролировать сроки обработки.

---

## Цель

Создать централизованную систему управления заявками на пропуск, которая:

- сокращает время обработки заявки;
- делает процесс прозрачным для всех участников;
- снижает количество ошибок;
- фиксирует историю изменения статусов;
- даёт основу для операционной и аналитической отчётности.

---

## Краткий обзор решения

Система включает следующие логические блоки:

- форма подачи заявки;
- backend-логика обработки;
- модуль согласования;
- управление статусами;
- уведомления участников процесса;
- хранение истории изменений.

Базовый поток процесса:

**Employee → Manager → Pass Administrator → Issued**

---

## Роли

- **Employee / Initiator** — создаёт заявку на пропуск;
- **Manager / Approver** — согласует или отклоняет заявку;
- **Pass Administrator / Security** — завершает обработку и выпускает пропуск;
- **Administrator** — техническая/административная роль в системе.

---

## Scope

В рамках кейса рассматриваются:

- создание заявки на пропуск;
- заполнение и валидация обязательных полей;
- назначение начального статуса;
- просмотр списка и карточки заявки;
- изменение текущего статуса;
- отклонение заявки с обязательным указанием причины;
- хранение истории смены статусов;
- базовая ролевая модель;
- SQL-запросы для проверки и анализа процесса;
- локальное развёртывание PostgreSQL через Docker.

## Модель данных

Текущая физическая реализация схемы описана в файле:

- [sql/schema.sql](./sql/schema.sql)

Основные сущности:

- `pass_request` — заявка на пропуск;
- `request_status_history` — история изменения статусов заявки;
- `status` — справочник статусов;
- `pass_type` — справочник типов пропусков;
- `app_user` — пользователи системы;
- `user_role` — роли пользователей.

Модель поддерживает:

- хранение текущего статуса заявки;
- фиксацию переходов между статусами;
- хранение причин отклонения;
- привязку заявки к инициатору;
- базовую аналитическую обработку через SQL.

---

## Что реализовано в SQL-части

В репозитории реализованы:

- схема PostgreSQL;
- тестовые данные;
- набор SQL-запросов для операционных и аналитических сценариев;
- индексы для основных сценариев поиска;
- триггер автоматического обновления `updated_at`;
- триггер, требующий `rejection_reason` при переводе заявки в статус `REJECTED`.

Примеры задач, которые можно проверить через SQL:
- список заявок с текущими статусами;
- отклонённые заявки и причины отклонения;
- история изменения статусов;
- количество заявок по статусам;
- анализ времени обработки заявки;
- проверка согласованности текущего статуса и истории.

Использовать те же параметры подключения:

Host: localhost
Port: 5432
Database: access_pass_db
Username: analyst
Password: analyst123
Проверочные SQL-запросы
Посмотреть заявки
SELECT *
FROM pass_request
LIMIT 10;
Посмотреть статусы
SELECT *
FROM status;
Количество заявок по текущим статусам
SELECT
    s.status_name,
    COUNT(*) AS request_count
FROM pass_request pr
JOIN status s ON pr.current_status_id = s.status_id
GROUP BY s.status_name
ORDER BY request_count DESC;
Отклонённые заявки
SELECT
    pr.request_number,
    pr.visitor_full_name,
    pr.rejection_reason
FROM pass_request pr
JOIN status s ON pr.current_status_id = s.status_id
WHERE s.status_code = 'REJECTED';
История изменений статусов
SELECT
    pr.request_number,
    s_from.status_name AS from_status,
    s_to.status_name AS to_status,
    rsh.changed_at
FROM request_status_history rsh
JOIN pass_request pr ON rsh.request_id = pr.request_id
LEFT JOIN status s_from ON rsh.from_status_id = s_from.status_id
LEFT JOIN status s_to ON rsh.to_status_id = s_to.status_id
ORDER BY rsh.changed_at DESC;
